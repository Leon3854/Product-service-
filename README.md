# 🚀 Distributed Product & Inventory System (NestJS + Go + Kafka)

Реализация масштабируемой микросервисной архитектуры для управления каталогом товаров и складскими остатками. Проект спроектирован с упором на **High Availability**, **Event-Driven Design** и **Cloud-Native** принципы.

# Итоги реализации (Что получилось на самом деле) читать в конце

## 🏗 Архитектурные особенности


1. **Polyglot Microservices:** Основная бизнес-логика реализована на **NestJS**, а высоконагруженный модуль складского учета (Inventory) — на **Golang** для минимизации задержек и потребления RAM.
2. **Event-Driven Communication:** Синхронизация данных между сервисами через **Apache Kafka**. Гарантированный порядок событий (Message Ordering) через использование Partition Keys.
3. **Resilience & Caching:** Распределенное кэширование в **Redis** с механизмом **Distributed Locking (SET NX)** для защиты от Race Conditions и Cache Stampede.
4. **Hybrid API:** Поддержка **REST** для внешних интеграций и **GraphQL** (Apollo) для гибких фронтенд-запросов (решена проблема N+1 через Prisma `include`).

### Итоги

В процессе разработки столкнулся с нюансами, которые не были известны на момент обдумывания ТЗ:

### 🛠 Что было сделано сверх плана:

Защита API (Rate Limiting): Изначально решил что в этом сервсие будет только бизнес логика и поэтому не планировал такую глубокую защиту, но похду работ читая советы понял, что открытый эндпоинт поиска — это «дыра» для ботов. Пришлось заниматья Sliding Window на Redis, который работает в распределенном режиме.
Оптимизация под Apple Silicon (M1): Пришлось пересобирать Docker-образы под архитектуру arm64 с сетевыми мостами Kafka, чтобы контейнеры видели друг друга. Теперь инфраструктура стабильна на любой платформе.
Умная инвалидация кэша: В ТЗ была просто очистка. На деле я внедрил точечный сброс (по ID и Slug), чтобы не «выбивать» из памяти Redis лишние данные.

### 🔄 Что пошло не по плану:

Проблема N+1 в GraphQL: При первых тестах вложенных запросов нагрузка на БД резко выросла. Пришлось решать вопрос с оптимизацией запросов через Prisma (Eager Loading), заложив фундамент для внедрения DataLoader.<br/>
Синхронизация Kafka: Сначала была идея обновлять всё в одном методе. Я полностью перешел на Event-Driven подход: теперь тяжелые задачи (счетчики, уведомления) решаются асинхронно. Увеличев скорость обработки до максиума(как я думаю).<br />
GitHub Container Registry сделан не до конца. Настрил фундамент в ci.yml но само хранилище (Registry) пока не «заселено». Сейчас робот просто собирает образ, проверяет, что он «не разваливается», и удаляет его.

### 🏗️ Polyglot Monorepo Architecture (NestJS + Go)


Проект спроектирован как распределенная система, где каждый сервис выбирает стек под конкретную задачу. Это позволяет сочетать скорость разработки на Node.js с экстремальной производительностью Golang.
🧩 Сервисы системы:
Product Service (NestJS / TypeScript):
Роль: Основная бизнес-логика, управление каталогом, сложный GraphQL API.
Почему NestJS: Высокая скорость итераций, отличная работа с I/O и богатая экосистема библиотек для валидации и кэширования.
Inventory Service (Golang):
Роль: Высоконагруженный складской учет и обработка транзакций в реальном времени.
Почему Go: Минимальное потребление памяти (в 10-15 раз меньше Node.js), нативная многопоточность и мгновенный запуск. Идеален для сервисов, требующих гарантированного времени отклика под нагрузкой.
🌉 Взаимодействие через Event Bus (Kafka):
Сервисы полностью изолированы друг от друга и общаются через события:
Когда Product Service создает товар, он публикует событие product.created в Kafka.
Inventory Service (Go) мгновенно подхватывает это событие через высокопроизводительный драйвер kafka-go, заводит карточку остатков и готовит склад к приемке.
Такой подход (Decoupling) гарантирует, что падение одного сервиса не остановит работу другого.

### На будущее план развития:
Для текущего микросервиса товаров BullMQ сейчас будет избыточен, но он идеально подйдет для следующего этапа развития системы.
Для текущих задач каталога товаров Kafka полностью закрывает потребности в асинхронном взаимодействии. Использование BullMQ на данном этапе было бы избыточным (Overengineering). Однако рассматривается BullMQ как инструмент для масштабирования фоновых задач, таких как генерация тяжелых отчетов или отложенная публикация товаров по расписанию, что легко интегрируется в текущую архитектуру через Redis
BullMQ — это не замена Kafka, это инструмент для тяжелых фоновых задач внутри самого сервиса. Его стоит «привинтить», если появятся такие задачи:
Генерация отчетов/прайс-листов: Когда пользователь просит выгрузить 100к товаров в Excel. Это нельзя делать в основном потоке, нужно поставить задачу в BullMQ.
Массовая обработка изображений: Если при загрузке товара нужно нарезать 5 видов превью (thumbnails).
Отложенные действия (Scheduled Jobs): Например, «опубликовать товар завтра в 09:00» или «снять скидку через 3 дня». BullMQ справляется с этим лучше, чем простые Cron-задачи.

Payment-Service — Глубокое погружение в финтех-логику: рекуррентные платежи, подписки и работа с очередями RabbitMQ.<br/>
WB-Service-Traffics — Важный этап осознанной паузы. Проект был приостановлен для детального изучения Golang и инфраструктурных паттернов, так как текущих знаний не хватало для реализации высоконагруженного трафика. <br/>
Current: Product-Service — Итоговый «эталонный шаблон» (Reference Architecture). Здесь объединены NestJS, Go, Kafka, Redis и Kubernetes. Проект служит фундаментом для всех будущих микросервисов системы.
